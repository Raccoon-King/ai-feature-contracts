# FC: Evaluate Prompt-Centric Agents vs Runtime-Driven Agents
**ID:** FC-005 | **Status:** draft
**Data Change:** yes
**API Change:** no
CONTRACT_TYPE: FEATURE_CONTRACT
ARCH_VERSION: v1
RULESET_VERSION: v1
ENV_VERSION: v1

## Objective

Evaluate the current prompt-centric agent model and define a migration path toward runtime-driven agents where contracts, tools, and execution code own deterministic workflow steps and the LLM is constrained to reasoning over structured context.

## Business Value

- **Predictability**: Reduce prompt drift by moving deterministic behavior into code
- **Efficiency**: Lower token usage by replacing repeated instructions with runtime-prepared context
- **Modularity**: Separate agent contract, tool orchestration, and LLM prompting into replaceable layers
- **Control**: Validate inputs, constrain outputs, and decide exactly what the LLM sees
- **Migration Safety**: Create a staged path instead of a disruptive rewrite

## Scope

### In Scope

- Assess how current agent YAML, workflows, prompt bundles, and runtime helpers divide responsibilities today
- Identify logic currently embedded in prompts that should move into runtime code or tool layers
- Map the target architecture into four layers: contract, runtime, tools, and minimal LLM prompt
- Evaluate existing building blocks in `agents/`, `workflows/`, and `lib/` that can support the shift
- Produce a gap analysis between the current prompt-centric model and a functioning runtime-driven agent model
- Recommend one pilot agent/workflow for migration and define staged implementation phases
- Produce example artifacts that show what a reduced prompt and runtime-oriented contract could look like

### Non-Goals

- Implementing the full agent runtime in this story
- Replacing all current agents or workflows
- Changing production CLI/API behavior as part of the evaluation
- Removing existing prompt-based handoff flows before a migration plan is approved
