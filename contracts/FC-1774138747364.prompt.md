## Mission

Audit the existing `grabby-contracts` repository, then implement the story plan in sequence so the rulesets become materially stronger for Node.js, Express, Jest, API compatibility, security, and Git workflow guidance.

The goal is not to dump generic rules into YAML. The goal is to produce a usable central rulesets repository with:

- actionable guidance
- concrete examples
- unique and stable rule IDs
- valid YAML and manifest structure
- documentation that explains how to author and maintain rulesets

## Hard Constraints

- Work in the target `grabby-contracts` repository, not in Grabby core.
- Do not change the sync protocol or invent new ruleset categories.
- Keep the work JavaScript/CommonJS oriented. Do not add TypeScript-only assumptions.
- Every new rule must include `id`, `name`, `severity`, `tags`, `description`, and `guidance`.
- Guidance must be practical, opinionated, and implementation-ready.
- Prefer ASCII in docs and YAML. Clean up mojibake or broken characters if you encounter them.
- Keep rule IDs unique across the repository.
- Do not rewrite unrelated rulesets just for style consistency.

## Required Working Style

Execute the work story by story in order. Do not skip ahead. Do not batch all edits into a single giant change before understanding the current repository state.

For each story:

1. Read the current file(s) first.
2. Compare the current content to the story acceptance criteria.
3. Implement only the scoped delta for that story.
4. Validate the changed YAML or docs before moving on.
5. Summarize what changed, what rule IDs were added, and whether the story is complete.
