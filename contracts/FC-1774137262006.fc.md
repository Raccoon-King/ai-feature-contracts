# FC: Separate Grabby engine from ruleset subsystem
**ID:** FC-1774137262006 | **Status:** approved
**Data Change:** no
**API Change:** no

## Ticket
- Who: Grabby maintainers and plugin developers
- What: Separate Grabby engine from ruleset subsystem into independent modules
- Why: Enable independent versioning, optional ruleset installation, cleaner architecture, and future plugin extensibility

## Objective
Decouple the Grabby core engine (CLI, contracts, workflows, personas) from the ruleset subsystem (sync, registry, builder) so they can be versioned, installed, and configured independently. The engine should run without ruleset modules present, and rulesets should load as an optional plugin.

## Background
Current coupling points identified:
1. **Config coupling** - `rulesets.*` section embedded in main `grabby.config.json`
2. **Command blocking** - `validate`, `plan`, `execute` call `performRulesetSyncCheck()` and can block
3. **Contract metadata** - Ruleset snapshots embedded in contract YAML frontmatter
4. **Shared utilities** - `manifest-parser.cjs`, `sync-lock.cjs` used by both systems

## Scope

### Phase 1: Config Separation
- Make `rulesets` config section optional with graceful fallback
- Create separate `rulesets.config.json` file option
- Engine config validates without ruleset schema

### Phase 2: Command Decoupling
- Remove sync check blocking from core commands (make advisory)
- Commands proceed if ruleset module not present
- Create hook system for ruleset checks to register

### Phase 3: Module Restructure
- Move ruleset files to `lib/rulesets/` subdirectory
- Organize: `sync.cjs`, `cli.cjs`, `builder.cjs`, `registry.cjs`, `authoring.cjs`
- Move shared utils to `lib/rulesets/common/`

### Phase 4: Plugin Architecture
- Create plugin registration system in `lib/plugins/`
- Rulesets register as first plugin
- Define plugin hooks: `onCommand`, `onContractCreate`, `onValidate`

### Phase 5: Contract Metadata Cleanup
- Make ruleset metadata in contracts optional
- Create separate `{contract-id}.rulesets.json` sidecar file option
- Migrate existing contracts gracefully

## Non-Goals
- Removing ruleset functionality entirely
- Breaking existing ruleset workflows
- Changing the external CLI interface
- Splitting into separate npm packages (yet)

## Directories
**Allowed:** `lib/`, `lib/rulesets/`, `lib/plugins/`, `tests/`, `bin/`
**Restricted:** `node_modules/`, `.git/`
