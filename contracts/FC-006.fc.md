# FC: Refresh Documentation and Website After FC-005
**ID:** FC-006 | **Status:** draft
**Data Change:** yes
**API Change:** yes
CONTRACT_TYPE: FEATURE_CONTRACT
ARCH_VERSION: v1
RULESET_VERSION: v1
ENV_VERSION: v1

## Objective

Fully update Grabby's documentation and website after `FC-005` is implemented so public-facing guidance, architecture docs, install flows, command references, and marketing copy consistently reflect the evaluated agent-runtime direction, the new mental model, and the current implementation status.

## Prerequisites

- `FC-005` is implemented and accepted
- Any architecture decisions or pilot outcomes from `FC-005` are finalized enough to document without speculative copy
- The owning implementation contract(s), if any, are identified so docs can distinguish shipped behavior from roadmap

## Business Value

- **Clarity**: Remove outdated prompt-centric explanations and replace them with the current operating model
- **Trust**: Keep docs and website aligned with what the product actually does
- **Adoption**: Make the runtime-driven model understandable to users evaluating Grabby
- **Supportability**: Reduce confusion across README, guides, website copy, and architecture documents
- **Release Readiness**: Ensure the repo meets its docs-and-website audit expectations before future release tagging

## Scope

### In Scope

- Audit docs and website content that describes prompts, personas, orchestration, agent execution, workflow control, and LLM responsibilities
- Update public-facing messaging to reflect the post-`FC-005` agent model and terminology
- Clarify what is implemented now, what is transitional, and what remains roadmap/future phase
- Refresh command references and walkthroughs where the runtime-driven model changes how users should think about the system
- Update architecture and protocol docs to explain the contract/runtime/tool/LLM separation
- Update the website landing page and docs landing page to match the new product framing
- Add a lightweight validation artifact or test that checks core docs/site references for consistency
- Produce a documentation audit checklist confirming docs and website were reviewed together

### Non-Goals

- Implementing the runtime architecture itself
- Rewriting unrelated docs that are not affected by `FC-005`
- Large visual redesign of the website beyond copy/structure needed for the new message
- Release tagging or publishing the site as part of this story

**Restricted:** `node_modules/`, `coverage/`, `.git/`, `.env*`, `.grabby/`, `lib/`, `bin/`, `agents/`, `workflows/`
